# Agenda and Notes

- [x] **effect producing operations within constructors**
  - William: I've noticed that minting/sending tokens in constructors does not work as expected with unshielded mints acting as a no-op and shielded mints/sends resulting in an insufficient funds error.
  - currently deployment results in constructor just initiating the state
  - so anything producing effect shouldn't be allowed in the constructor (Parisa: expand on issue 117 and include kernel.self)
- [x] **Hardfork and multi-env discussion follow up**
  - execution context constructor is releaved from knowing the structure of contract state
  - contract state is defined in onchain runtime
  - so future ocrt have to ensure future compatibility (pieces of contract state required to construct exection context)
  - so on compiler side, the user can just use a compiler pragma to dictate the version used
- [x] **Simplifying `n as Field` where `n` is a numeric literal**
  - some literal syntax (not easy to scale):
    - from rust: `f_123` `123_f` (Kevin: we shouldn't just grab from different langs)
    - follow TS convention `12n` 
    - `0F123`
  - determine the exact type at call site and introduce an internal literal numeric type 
  - you have to write `123 as Feild` (Kevin: stay on this until we hear from users. Attendees agreed. Any objections from other TSC members?)
    - add a coip to capture the discussion so we won't keep coming back to it.
